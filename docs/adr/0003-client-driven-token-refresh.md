# ADR-0003: Обновление токенов инициирует клиент, не API

- **Статус:** Accepted
- **Дата:** 2026-07-16

## Контекст

Аутентификация построена на коротких access-JWT и refresh-токенах с ротацией.
Возник вопрос: кто выполняет обновление access-токена при его истечении — API
(прозрачно, внутри middleware) или клиент.

Ранее gateway обновлял токен «тихо» прямо в identity-middleware: при истёкшем
access-токене он сам дёргал refresh и подменял cookie. Это неявное поведение
(«магия»), размазанное по каждому защищённому эндпоинту, и оно противоречит
принципу `engineering-principles.md` «Явное лучше неявного».

Важно различать две ответственности: **проверка** валидности и срока access-токена
— обязанность API на каждом запросе (JWT самодостаточен, проверка stateless и
дешёвая); **инициирование обновления** — это оркестрация, и её место у клиента.

## Рассмотренные варианты

- **Тихий refresh на gateway** (прежнее). Клиент ничего не делает, но обновление
  скрыто в каждом эндпоинте, поведение неявное, контракт непрозрачен.
- **401 + явный endpoint, refresh инициирует клиент (выбрано).** Стандарт
  индустрии: API возвращает 401 с признаком истечения; клиент по интерсептору
  зовёт отдельный refresh и повторяет запрос.

## Решение

- На каждом защищённом запросе gateway проверяет подпись и `exp` access-токена.
- Если access-токен истёк — gateway возвращает **401** с телом
  `{"code":"token_expired", ...}` и **не** обновляет токен неявно.
- Обновление вынесено в явный эндпоинт `POST /api/auth/refresh`, доступный без
  прохождения identity-middleware.
- Клиент по 401 c `code=token_expired` зовёт `/auth/refresh` (refresh-cookie
  уходит автоматически), получает новые токены и **один раз** повторяет исходный
  запрос; при неуспехе refresh — уходит на логин.
- Refresh-токен остаётся stateful (хранится и ротируется на стороне auth-сервиса).

## Последствия

- Положительные: явный, предсказуемый, стандартный контракт; нет скрытых
  обновлений внутри бизнес-эндпоинтов; поток отлаживается и тестируется.
- Отрицательные: клиент обязан реализовать интерсептор (перехват 401 → refresh →
  повтор). Это разовая, понятная работа на стороне клиента.
- Критерии пересмотра: переход на другой механизм сессий (например, полностью
  stateless без серверной ротации) — новый ADR.
