# ADR-0000: Вести Architecture Decision Records

- **Статус:** Accepted
- **Дата:** 2026-07-14

## Контекст

`docs/project-strategy.md` и `docs/engineering-principles.md` требуют, чтобы
каждое значимое архитектурное решение фиксировалось: контекст, альтернативы,
выбор, последствия. Без единого места для таких записей обоснования решений
живут в чужой памяти и теряются — через год ответ на вопрос «почему здесь так?»
приходится реконструировать по коду.

## Рассмотренные варианты

- **ADR-файлы в репозитории** — лёгкие Markdown-записи рядом с кодом,
  версионируются вместе с ним, доступны в code review и офлайн.
- **Внешняя вики / трекер** — решения в Confluence/Notion/issues. Отрываются от
  кода, устаревают, недоступны при работе с репозиторием без сети.
- **Не фиксировать** — противоречит стратегии; знание теряется.

## Решение

Вести ADR в каталоге `docs/adr/` в формате Nygard (см. `template.md`). Каждое
значимое решение сопровождается ADR в том же изменении, что и реализация. ADR
неизменяем; смена решения оформляется новым ADR со ссылкой `Superseded by`.

## Последствия

- Положительные: единый прослеживаемый журнал решений; предсказуемость;
  выполнение требования стратегии о документировании решений.
- Отрицательные: небольшая дисциплинарная нагрузка — значимое решение требует
  ещё и записи. Это осознанная и оплаченная цена.
- Критерии пересмотра: не предполагается; отказ от ADR сам был бы ADR.
