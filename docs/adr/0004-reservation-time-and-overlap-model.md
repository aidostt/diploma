# ADR-0004: Модель времени брони и защита от двойного бронирования

- **Статус:** Accepted (реализация ожидает публикации proto-контракта и БД)
- **Дата:** 2026-07-16

## Контекст

Текущая модель брони некорректна на уровне домена:

- `reservationtime` — свободный текст («1:00 PM»); нормализации нет.
- `reservationdate` захардкожен в `CURRENT_DATE` при вставке → бронь на будущую
  дату невозможна, а сравнение занятости игнорирует дату.
- Длительности брони нет вообще → понятие «пересекающийся интервал» (ключевой
  инвариант из `engineering-principles.md`) невыразимо.
- Занятость стола хранится одним булевым флагом `restables.isreserved`, что
  несовместимо со слотами: стол не может быть «занят в 19:00 и свободен в 21:00».
- Проверка занятости — `check-then-insert` без constraint → гонка → двойная бронь.

Отдельно обсуждался риск злоупотребления: клиент, задающий длительность сам,
может «занять всё время».

## Решение

**Время брони моделируется интервалом, длительность задаёт заведение, а не клиент.**

1. **Контракт:** клиент присылает `start_at` (timestamptz) и `party_size` — и
   **не** присылает длительность или `end`. Длительность (turn time) — политика
   заведения (конфиг; расширяемо до зависимости от размера компании, напр. 2
   гостя — 90 мин, 6 — 150 мин).
2. **Схема (PostgreSQL):**
   ```sql
   CREATE EXTENSION IF NOT EXISTS btree_gist;
   -- start_at timestamptz NOT NULL; duration берётся из политики заведения
   ALTER TABLE reservations
     ADD COLUMN period tstzrange
       GENERATED ALWAYS AS (tstzrange(start_at, start_at + turn_interval)) STORED,
     ADD CONSTRAINT reservations_no_overlap
       EXCLUDE USING gist (tableid WITH =, period WITH &&);
   ```
   Инвариант «нет двух броней одного стола с пересекающимися интервалами»
   гарантируется БД (race-free, индексируется gist). Нарушение (`23P01`) →
   `codes.AlreadyExists` → HTTP 409.
3. **Убрать `isreserved` как источник истины.** Доступность стола = «нет брони с
   пересекающимся `period`»; булев флаг удаляется.
4. **Вместимость:** `party_size` сопоставляется с `restables.numberofseats`.
5. **Владение (закрытие reservation-IDOR):** идентичность вызывающего
   (`user_id`, `roles`) пробрасывается в reservation-сервис через gRPC-метадату
   и извлекается серверным интерцептором; доступ к брони — владелец или персонал
   (`admin`/`restaurantAdmin`/`waiter`), иначе `PermissionDenied`. Так enforcement
   живёт в домене, а не размазан по gateway (без TOCTOU).
6. **Анти-абьюз (отдельный слой, не модель времени):** лимит активных броней на
   пользователя, rate-limiting на создание брони, верификация аккаунта; депозит /
   hold карты против no-show — как в реальных ресторанах. Фиксируется как
   направление, реализуется отдельными шагами.

## Последствия

- Положительные: корректность и конкурентная безопасность инварианта решены на
  уровне БД; появляется будущее бронирование, реальные интервалы, вместимость;
  IDOR закрывается в домене; злоупотребление длительностью исключено (длительность
  серверная).
- Отрицательные / стоимость реализации:
  - **Меняется proto-контракт** (`reservation.proto`: `reservationTime` →
    `start_at` + `party_size`), а сервисы зависят от **опубликованной** версии
    `github.com/aidostt/protos` (сейчас ещё и рассинхрон версий: auth `v0.5.6`,
    qr `v0.5.7`, прочие `v0.6.5`). Значит нужно: регенерация → тег/публикация
    новой версии → синхронный bump во всех сервисах.
  - Нужна миграция данных и прогон против реального PostgreSQL (`btree_gist`,
    `EXCLUDE`) — проверяется в CI/локальном Docker.
- Критерии пересмотра: если понадобится модель без фиксированной длительности
  (например, поминутная аренда) — новый ADR; базовый интервально-constraint-подход
  при этом сохраняется.

## Статус реализации

**Ядро реализовано и проверено** (protos `v0.7.0`; интервальная модель
`start_at`/`ends_at`/`period` + `EXCLUDE USING gist`; проверка вместимости;
`23P01` → 409; все сервисы синхронизированы на `v0.7.0`). Инвариант подтверждён
integration-тестом (testcontainers) против реального PostgreSQL: пересекающаяся
бронь отклоняется, смежная — проходит.

Осталось отдельными шагами (не блокирует ядро):

- **Reservation ownership (IDOR):** enforcement владения в сервисе через
  gRPC-метадату + серверный интерцептор (без изменения сообщений proto).
- **Доступность столов:** убрать `restables.isreserved` как источник истины;
  `GetAvailableTables`/`GetReservedTables` — как запросы «свободен в интервале»
  (требует добавления `start_at` в `table.proto`).
- **Анти-абьюз:** лимит активных броней на пользователя, rate-limiting, депозит.
